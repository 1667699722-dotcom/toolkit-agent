# 通用具身智能物理运动系统技术报告

## 摘要

本文档提出一套分层解耦的具身智能物理运动控制系统架构，适用于**四足机器人、双足机器人、轮式无人车、履带式无人平台**等一系列无人自主运动产品。

该架构通过**两模型分层设计**（大语言模型 LLM 负责语义理解与任务调度，专门训练的具身视觉-动作基础模型 Embodied Vision-Action Model 负责在物理真实场景下决策），结合**动作 token 化与运动基元库**，实现了从自然语言指令到物理运动的完整闭环，并通过**结构先验翻译层**保证了系统的稳定性与可解释性。

---

## 1. 问题背景与设计目标

### 1.1 现状与挑战

当前无人自主运动产品普遍存在以下痛点：

1. **调参地狱**：传统方案依赖手写步态生成器、MPC 控制器、PID 参数拟合，不同场景、地形需要重新调参
2. **泛化性差**：手写控制器难以适应新地形、新任务
3. **人机交互不友好**：用户需要学习命令格式或遥控器操作，而不是用自然语言指令
4. **通用 VLM 不适用**：GPT-4o、Claude 3.5 等通用视觉-语言模型对机器人物理运动、稳定性边界、关节扭矩限制等缺乏理解，无法直接用于生成可靠的运动决策
5. **端到端训练门槛高**：训练专门的具身基础模型需要海量仿真/真实机器人数据和大规模 GPU 集群，只有少数大公司能负担

### 1.2 设计目标

本架构旨在解决上述问题，实现：

| 目标 | 描述 |
|------|------|
| **自然语言交互** | 用户可以直接用自然语言下达任务 |
| **场景泛化** | 基于专门训练的具身模型，能适应新地形、新任务 |
| **可解释性** | 每一层的决策都可以被追踪和理解 |
| **安全可靠** | 内置多层安全防护，避免不合理运动导致设备损坏或安全事故 |
| **跨平台适配** | 同一架构可以适配四足、双足、轮式、履带式等不同运动平台 |
| **渐进式开发** | 可以先用监督学习训练小规模具身模型，再逐步扩展 |

## 2. 总体架构设计

### 2.1 系统分层图

```
用户自然语言指令
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│ 层 1：大语言模型（LLM）调度层（事件驱动，<1Hz）                │
│ 功能：意图理解 → 任务分解 → 工具调用                           │
│ 工具：                                                         │
│    - run_task(task_description, options)：启动持续任务       │
│    - stop_task(reason)：紧急停止                              │
│    - query_status()：查询当前状态                             │
└──────────────────────────┬────────────────────────────────────┘
                           │ tool call（JSON 格式任务描述）
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 层 2：具身视觉-动作基础模型（Embodied VAM）执行层（5-10Hz）    │
│ 核心：在大量仿真/真实机器人数据上专门训练，不是通用 VLM          │
│ 功能：任务持锁 → 多模态感知（图像 + 关节状态 + IMU）            │
│        → 动作 token 生成 → 完成判定                           │
│ 输入：                                                         │
│    - 相机 RGB/深度图像                                         │
│    - 任务描述（循环过程中保持不变）                             │
│    - 当前平台全状态（关节角度/速度、身体高度/位姿、接触力）      │
│ 输出：                                                         │
│    - action_tokens：结构化运动参数（跨平台通用语义 + 平台特化） │
│    - done：任务是否完成（true/false）                          │
└──────────────────────────┬────────────────────────────────────┘
                           │ 动作 token（JSON 格式，10-30 维语义参数）
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 层 3：动作 token 翻译层（平台特化，200-1000Hz）                │
│ 功能：语义参数 → 平台运动学 → 执行器目标信号                   │
│ 输入：action_tokens + 当前时间/相位                           │
│ 输出：目标关节角度 / 目标轮速 / 目标控制信号                   │
│ 频率：由底层执行器的控制周期决定                               │
└──────────────────────────┬────────────────────────────────────┘
                           │ 目标执行信号
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 层 4：物理执行层（200-10000Hz）                                │
│ 功能：PD 控制 / 力矩控制 → 电机/执行器 → 物理运动               │
│ 技术：电机驱动、减速器、传感器反馈、物理约束（关节限位、扭矩限幅）│
└────────────────────────────────────────────────────────────────┘
```

### 2.2 关键设计原则

| 原则 | 说明 |
|------|------|
| **分层解耦，各司其职** | 每一层只负责自己最擅长的事情，层间通过标准化接口通信 |
| **具身模型专门训练** | 层 2 必须是在机器人物理数据上专门训练的具身基础模型，不能用通用 VLM |
| **动作 token 化，而非原始信号** | 具身模型不直接输出关节角度或电机电流，而是输出有物理意义的结构化动作参数 |
| **平台无关语义，平台特化翻译** | 语义参数（如“前进速度”、“转向角速度”）是跨平台通用的，翻译层负责适配不同平台的运动学 |
| **结构先验编码安全约束** | 翻译层内置关节限位、扭矩限幅、物理稳定性约束等先验知识，保证安全 |
| **事件驱动，避免无效轮询** | LLM 不持续轮询，只在任务开始/完成/出错时被唤醒 |

## 3. 各层详细技术设计

### 3.1 层 1：LLM 调度层

#### 3.1.1 核心职责

1. **自然语言意图理解**：从用户自然语言中提取任务目标、约束条件、优先级等
2. **任务分解与调度**：复杂任务分解为一系列子任务
3. **工具调用与结果处理**：通过 function calling 调用执行层，处理返回结果
4. **上下文管理**：维护对话历史、任务状态、环境上下文

#### 3.1.2 典型交互流程

```
用户: "把客厅的垃圾倒掉，然后回到门口等待"
│
├─▶ LLM 推理
│   ├─ 意图：清理任务 + 返回任务
│   ├─ 子任务分解：
│   │   1. 移动到客厅垃圾桶位置
│   │   2. 抓取垃圾袋
│   │   3. 移动到门口垃圾桶
│   │   4. 放入垃圾袋
│   │   5. 返回门口
│   │   6. 等待
│
└─▶ tool call: run_task("移动到客厅垃圾桶位置")
    │
    └─▶ (执行中...)
          │
          └─▶ (执行完成)
          │
          └─▶ LLM 继续推理，调用下一个子任务
```

#### 3.1.3 工具定义示例（OpenAI Function Call 格式）

```json
{
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "run_task",

        "parameters": {
          "type": "object",
          "properties": {
            "task_description": {
              "type": "string",
              "description": "用自然语言描述要执行的任务，例如：'移动到客厅垃圾桶位置'"
            },
            "options": {
              "type": "object",
              "description": "可选任务参数",
              "properties": {
                "max_duration": {"type": "number", "description": "最大执行时间（秒）"},
                "priority": {"type": "string", "enum": ["low", "normal", "high"], "description": "任务优先级"},
                "terrain_hint": {"type": "string", "description": "地形提示，帮助平台选择合适的运动模式"}
              }
            }
          },
          "required": ["task_description"]
        }
      }
    },
    {
      "type": "function",
      "function": {
        "name": "stop_task",
        "description": "立即停止当前正在执行的任务",
        "parameters": {
          "type": "object",
          "properties": {
            "reason": {"type": "string", "description": "停止原因"}
          }
        }
      }
    }
  ]
}
```

### 3.2 层 2：具身视觉-动作基础模型（Embodied VAM）执行层

#### 3.2.1 核心设计思想

具身视觉-动作基础模型（Embodied Vision-Action Model）是整个系统的核心，**必须在大规模仿真/真实机器人数据上专门训练**。

这一层不直接输出底层执行信号，而是输出**跨平台通用的语义动作参数**（动作 token），通过翻译层适配不同平台。

#### 3.2.2 具身模型训练数据来源

| 数据类型 | 说明 |
|----------|------|
| **仿真数据**（主要来源） | 在 Isaac Gym、MuJoCo 等并行仿真器里跑数千万步，生成「多模态感知 → 运动决策」配对数据 |
| **真实机器人数据** | 用多台真实机器在真实环境中记录轨迹，做 Sim-to-Real 微调 |
| **混合数据** | 仿真数据为主，真实数据为辅，共同训练提高泛化性 |

#### 3.2.3 典型的具身基础模型

| 模型 | 机构 | 训练方式 | 特点 |
|------|------|----------|------|
| **GR00T** | NVIDIA | Isaac Gym 4096 并行仿真器，大规模自监督训练 | 通用具身基础模型，支持多种机器人形态 |
| **RoboCat** | Google DeepMind | 真实机器人 + 仿真混合训练 | 在多种机械臂任务上泛化 |
| **RT-2 / RT-X** | Google DeepMind | 在真实机器人数据上微调视觉-语言模型 | 通用机器人基础模型 |
| **Octo** | Stanford/UC Berkeley | 在 100 万步机器人数据上训练的开源模型 | 小团队/个人可用，已预训练 |

#### 3.2.4 具身模型输出格式（动作 token 规范）

输出严格遵循 JSON 格式，动作 token 分为三部分：

1. **运动模式与步态参数**（通用）
2. **身体状态目标**（平台特化）
3. **运动速度指令**（通用）

```json
{
  "action_tokens": {
    "motion_mode": {
      "type": "walk", // 或 trot / crawl / climb / roll / drive 等平台支持的模式
      "amplitude": 0.2, // 摆动/步幅幅度
      "frequency": 2.0, // 步态/动作频率
      "lift_clearance": 0.15, // 抬脚/离地高度
      "duty_cycle": 0.5 // 支撑占比
    },
    "body_state": {
      "height": 0.26, // 目标身体高度
      "pitch": 0.0, // 目标俯仰角
      "roll": 0.0, // 目标横滚角
      "yaw_rate": 0.0 // 目标横摆角速度
    },
    "velocity": {
      "forward": 0.5, // 目标前进速度
      "lateral": 0.0, // 目标侧向速度
      "yaw": 0.0 // 目标转向角速度
    }
  },
  "done": false
}
```

### 3.3 层 3：动作 token 翻译层

#### 3.3.1 核心定位

翻译层是整个系统中**唯一需要针对具体平台进行特化开发**的部分，它把具身模型输出的跨平台通用语义参数转换为特定平台的执行器目标信号。

翻译层的核心是**运动基元（motion primitive）库** + **纯几何/运动学计算**，不包含复杂的闭环控制逻辑（闭环控制由物理执行层的 PD 或力矩控制负责）。

翻译层内置以下**结构先验**知识：
- 平台运动学模型
- 关节限位、扭矩限幅等物理约束
- 步态/运动模式的相位同步关系
- 稳定性边界

#### 3.3.2 翻译层通用架构

翻译层分为三个模块：

```
action_tokens（语义参数）
         │
         ▼
┌───────────────────────────────────┐
│ 1. 参数范围校验与钳制模块          │
│  - 限制语义参数到物理安全范围     │
│  - 处理模型输出的不合理值        │
└───────────────────┬────────────────┘
                    │ 校验后的参数
                    ▼
┌───────────────────────────────────┐
│ 2. 运动基元与相位生成模块          │
│  - 根据 motion_mode 选择运动基元  │
│  - 根据 frequency 推进相位        │
│  - 计算当前相位下的各执行器轨迹    │
└───────────────────┬────────────────┘
                    │ 轨迹/目标信号
                    ▼
┌───────────────────────────────────┐
│ 3. 运动学逆解与限幅模块            │
│  - 根据轨迹计算目标关节角度/轮速  │
│  - 应用关节限位、扭矩限幅          │
└───────────────────┬────────────────┘
                    │ 最终目标执行信号
                    ▼
输出到物理执行层
```

### 3.4 层 4：物理执行层

#### 3.4.1 PD 位置控制（主流方案）

主流无人运动产品使用 PD 位置控制：

```
tau = KP * (q_des - q_meas) + KD * (dq_des - dq_meas)
```

其中：
- `tau`：电机输出扭矩
- `KP`：位置增益矩阵
- `KD`：速度增益矩阵
- `q_des` / `dq_des`：目标关节角度/角速度（来自翻译层）
- `q_meas` / `dq_meas`：实测关节角度/角速度（来自编码器/IMU）

PD 增益通常是**平台级固定参数**，不需要随任务动态调整，这保证了系统的稳定性。

#### 3.4.2 物理约束与安全

物理执行层是最后一道安全防线，必须内置：
- **关节硬限位**：基于平台机械结构的物理限制
- **扭矩/电流限幅**：基于电机和驱动器额定值
- **硬件急停**：独立于所有软件层的物理急停按钮/开关
- **通信超时保护**：层间通信超时后自动进入安全停止模式

## 4. 安全设计多层防护

本架构采用**多层纵深防御策略**保证系统安全：

### 4.1 层 1（LLM）安全防护

- **工具调用权限控制**：严格限制工具调用范围，禁止调用未授权功能
- **任务合理性校验**：LMM 内部先校验任务是否合理，是否存在安全风险
- **用户确认机制**（可选）：高危任务需要用户二次确认

### 4.2 层 2（具身模型）安全防护

- **训练中学习安全边界**：在数据中加入不安全场景的负样本，让模型学会避免危险
- **感知异常检测**：模型内部检测到危险场景（如楼梯边缘、悬崖）主动停止
- **输出约束**：模型训练时就加入动作参数约束，避免超范围输出

### 4.3 层 3（翻译层）安全防护

- **参数范围硬钳制**：不管模型输出什么，翻译层都会把参数限制在物理安全范围内
- **关节限位**：确保目标关节角度永远不会超出平台机械结构限制
- **运动学可行性校验**：拒绝不可行的运动学请求

### 4.4 层 4（物理执行层）安全防护

- **扭矩/电流硬限幅**：超过额定值直接钳制
- **硬件急停**：独立的物理安全机制
- **通信超时保护**：自动安全停止

## 5. 跨平台适配思路

同一架构可以适配不同运动平台，核心是更换**翻译层**，并使用对应平台的数据**微调具身基础模型**：

| 平台类型 | 关键运动模式 | 翻译层主要工作 | 具身模型适配方式 |
|----------|--------------|----------------|-------------------|
| 四足机器人 | walk / trot / crawl / stand | 对角 trot 相位同步、腿摆动轨迹生成、逆运动学 | 在四足仿真/真实数据上微调预训练模型（如 Octo） |
| 双足机器人 | stand / walk / run | ZMP 轨迹跟踪、支撑脚摆动相位同步、逆运动学 | 在双足机器人数据上微调 |
| 轮式无人车 | drive / brake / spin | 速度指令转成左右轮速差、阿克曼转向角计算 | 在轮式车数据上微调 |
| 履带式平台 | drive / pivot_turn / spin_in_place | 速度指令转成左右履带差速 | 在履带平台数据上微调 |

## 6. 设计考量

### 6.1 为什么通用 VLM 不适用，必须用专门训练的具身模型？

通用 VLM（GPT-4o、Claude 3.5）的问题：
1. **不懂物理运动**：对机器人关节、扭矩、稳定性边界等完全没有概念
2. **没有真实反馈经验**：不知道什么运动模式会导致狗摔倒
3. **推理速度慢，不稳定**：每次推理都是重新理解，没有时序连贯性
4. **不可靠的 JSON 输出**：通用 VLM 经常输出格式错误的 JSON，不适合硬实时控制

专门训练的具身模型的优势：
1. **懂物理**：在训练数据中学习了机器人物理特性
2. **有时序记忆**：能记住前几步的状态，做出连贯决策
3. **输出稳定可靠**：训练时就约束了输出格式和范围
4. **推理速度快**：小模型，专门优化，能满足 5-10Hz 控制频率

### 6.2 为什么具身模型不直接输出原始执行信号？

问题分析：
1. **空间维度太大**：12 个关节角度是高维连续空间，难以稳定输出
2. **相位同步依赖硬约束**：四足对角 trot 等运动需要腿间严格相位同步，靠模型隐式学习不稳定
3. **推理速度无法满足控制频率**：具身模型推理 100-200ms，无法满足 200Hz 控制频率

解决方案：动作 token 化 + 翻译层，把高维控制问题拆解为低维语义决策（具身模型）和高频率相位生成（翻译层）。

### 6.3 为什么保留翻译层而不是完全端到端？

翻译层是**结构先验的载体**，它编码了人类对机器人运动的先验知识（步态规律、物理约束、运动学关系）。

保留翻译层的好处：
- **大幅降低训练数据需求**：不需要从头学走路
- **保证运动稳定性**：相位同步等硬约束由翻译层保证
- **提高可解释性**：可以追踪从语义参数到关节角度的完整计算过程
- **简化模型任务**：具身模型只需要做高层决策，不需要关心底层物理实现细节

### 6.4 为什么是分层架构而不是单模型端到端？

分层架构的优势：
- **每一层可以独立迭代升级**：步态生成换 RL，具身模型换更好架构，互不影响
- **每一层可以用最合适的技术**：LLM 做语义，具身模型做视觉-动作，传统 PD 做控制
- **易于调试与排错**：可以单独测试每一层，快速定位问题
- **安全性更好**：多层防护，某一层出错不会导致灾难
